Container Optimization
& Performance

Containerfile Best Practices

Key Principle: Least-changing content at the bottom
  • Order layers by change frequency
  • System dependencies first
  • Application code last

FROM ubi9:latest

RUN dnf install -y python3

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . /app

Chunkah: Context-Aware Layer Splitting

chunkah - Intelligent Dockerfile layer optimization

  • Post-build layer optimization tool
  • Context-aware splitting of content into optimal layers
  • Works seamlessly with zstd:chunked compression
  • Reduces duplicate content across image layers
Workflow: Build normally → Run chunkah → Push optimized image

https://github.com/coreos/chunkah

zstd:chunked Compression

Benefits:
  • Superior compression
  • Faster image pulls
  • Reduced storage costs
  • Better for CI/CD
Configuration: 

# Podman push
podman push \
  --compression-format zstd:chunked \
  myimage:latest

Note: Verify your registry supports zstd:chunked

Skaffold: Compose to Kubernetes


# Convert Docker Compose to Kubernetes manifests
skaffold init --compose-file docker-compose.yml

# Development with hot-reload
skaffold dev

# Build and deploy
skaffold run
  • Automatically converts Compose files to K8s manifests
  • Supports hot-reload for rapid development
  • Integrates with existing CI/CD pipelines
  • Maintains developer experience

https://skaffold.dev

https://www.sigstore.dev

Git Signing with Gitsign


# Configure Git to use gitsign
git config --global gpg.format x509
git config --global gpg.x509.program gitsign
git config --global commit.gpgsign true

# Commits are automatically signed via OIDC
git commit -m "feat: add new feature"

# Verify signature
git verify-commit HEAD
DevSpaces Integration: Enable tas-git-config extension to automatically configure gitsign in DevSpaces environments

Container Signing with Cosign


# Sign a container image
cosign sign registry.example.com/myapp:v1.0.0

# Verify signature
cosign verify \
  --certificate-identity=user@example.com \
  --certificate-oidc-issuer=https://github.com/login/oauth \
  registry.example.com/myapp:v1.0.0

# Attach attestations (SBOM, scan results)
cosign attest --predicate sbom.json \
  registry.example.com/myapp:v1.0.0
  • Sign images after build
  • Attach SBOMs and scan results
  • Verify before deployment

Security
Enforcement

Policy-Based Security Controls

  • Signature Verification: Only pull signed images
  • Vulnerability Thresholds: Block images with critical CVEs
  • License Compliance: Enforce approved licenses
  • Configuration Standards: Validate K8s manifests
Enforcement Points:
  • CI/CD pipeline gates
  • Admission controllers (Kubernetes)
  • Registry pull policies
  • Runtime security (Rancher, OpenShift)

Image Scanning in Rancher

  • Integrated vulnerability scanning
  • Continuous monitoring of running workloads
  • Policy-based alerts and blocking
  • Compliance reporting

# Example: Block images with HIGH/CRITICAL vulnerabilities
apiVersion: security.rancher.io/v1
kind: ImageScanPolicy
metadata:
  name: block-critical
spec:
  maxCritical: 0
  maxHigh: 0
  action: block

vLLM: CPU-Only Mode

  • Run large language models without GPUs
  • Testing and development on standard hardware
  • Available in Rancher deployments
  • Reduced infrastructure costs for non-production

# Deploy vLLM in CPU mode
helm install vllm-cpu rancher/vllm \
  --set model=meta-llama/Llama-2-7b \
  --set device=cpu \
  --set replicas=1

Note: CPU inference is slower but suitable for development/testing

Kaiden: Sandboxed Coding Agents

  • Run AI coding agents in secure sandboxes

  • Prevent unintended system modifications

Safety First: AI-generated code executes in isolated containers, preventing potential security or stability issues

Modern Development
Workflows

Evolution: Amadeus Case Study

Four-Stage Journey

  1. Vagrant + Docker + OpenShift: Fast but hard to support VMs
  2. Docker Desktop + Kind: Easy setup, no file sync, not true OpenShift
  3. Podman Desktop + Kind: Better container engine, similar workflow
  4. Podman Desktop + IDE + Helm: Local development with K8s testing, CI/CD on real cluster
Current State: Developers use Podman Desktop locally, Helm/Kind for local K8s testing, full CI/CD runs on production-like clusters

Hybrid Development Pattern

Local Development

  • Podman Desktop
  • IDE integration
  • Fast iteration cycles
  • Compose for multi-container

Testing & CI/CD

  • Kind for local K8s
  • Helm charts
  • Full cluster testing in CI
  • Production parity

Key Takeaways

  • Container Optimization: Use zstd:chunked and chunkah for faster pulls
  • Kubernetes Migration: Adopt gradually with Compose → Skaffold → Zarf
  • Security: Implement code signing and scanning early in the pipeline
  • Edge Computing: Immutable OS + Edge Manager for reliable deployments
  • Developer Experience: Balance local speed with production parity
  • AI Integration: Sandbox AI-generated code, test LLMs on CPU

Implementation Priorities

Suggested Adoption Order

  1. Quick Wins: Container optimization (chunkah, zstd:chunked)
  2. Security Foundation: Image scanning and vulnerability policies
  3. Developer Platform: RHDH for standardized onboarding
  4. Kubernetes Strategy: Pilot Skaffold for integration testing
  5. Supply Chain: Implement Sigstore-based signing
  6. Edge (if applicable): Evaluate MicroShift or Device Edge
